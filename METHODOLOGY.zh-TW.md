# 從散亂素材到可用知識：AI 輔助知識編譯方法論

你有一堆散落的內容——貼文、回覆、課程逐字稿、筆記、截圖、錄音——你知道裡面有價值，但每次要用的時候找不到，找到了也要重新讀一遍才能想起來在講什麼。

這份文件記錄一套經過實戰驗證的流程：**用 AI 把任何格式的原始素材，先提煉成結構化的知識原子，再編譯成可閱讀、可交叉引用的 Wiki 頁面。**

不是 RAG。不是每次問問題都重新搜尋原始資料。是一次性投入 token 把知識整理好，之後直接翻 Wiki。

---

## 核心論點：Compile > RAG（在個人知識庫的尺度下）

Andrej Karpathy 在 2025 年推的 LLM Wiki 模式，核心主張是「編譯」優於「檢索增強生成」：

**RAG 的問題**：每次查詢都要從原始資料重新搜尋、重新理解、重新組合。Token 花在重複的理解上，而且每次結果可能不一樣。

**Compile 的做法**：花一次 token 把原始資料理解透，產出結構化的 Wiki 頁面。之後查詢只要翻 Wiki，不用碰原始資料。知識是「持久的、複利的產物」，不是每次即時生成的。

### 這個論點成立的前提

1. **知識量在可管理範圍**——100-200 篇 Wiki 頁面以內，LLM 翻 index 就能定位。超過這個規模，可能需要混合方案（Wiki 當主結構 + 向量搜尋輔助定位）
2. **知識相對穩定**——你整理的是「認知地圖」不是「即時新聞」。一週更新幾次，不是每分鐘都在變
3. **有明確的主人**——個人知識庫有統一的觀點和立場，不是要聚合一百個人的不同意見
4. **品質比覆蓋率重要**——寧可 50 頁寫透的 Wiki，也不要 500 頁淺淺帶過的索引

### 誠實的限制

- 沒有同行評審的基準測試比較 Compile vs RAG 的效果。Karpathy 自己的證據是「用起來很爽」
- 超過幾百篇筆記時，純 Wiki 結構會碰到導航瓶頸。建議搭配向量搜尋做定位
- 編譯成本不低——首次 Ingest 大量素材需要可觀的 token。但之後的邊際成本遞減
- 知識會過時。需要定期 Lint（見下文），不是編譯完就不管了

## 流程總覽

```
原始素材（任何格式的文字）
    ↓ Phase 1：設計分類結構
    ↓ Phase 2：段落分類（值得萃取 / 跳過）
    ↓ Phase 3：萃取成知識原子（每個一個核心論點）
結構化知識原子
    ↓ Phase 4：品質整理（去重、分類校準、缺口分析）
    ↓ Phase 5：外部校驗 + 分支統整
經過驗證的原子 + 分支摘要
    ↓ Phase 6：編譯成 Wiki 頁面
Wiki 頁面（可閱讀、有交叉引用）
    ↓ 持續維護
Ingest 新素材 / Query 回寫 / Lint 檢查
```

以下每個 Phase 都可以獨立執行。你可以只做到原子就停，也可以跳過原子直接編譯 Wiki。但完整走過一遍的效果最好。

---

## Phase 1：骨架設計

**目標**：決定知識庫的主題樹——你的知識要怎麼分類。

### 輸入
- 你對自己知識版圖的直覺
- （選填）你的教學產品、內容計劃、工作領域

### 操作

1. 列出你的知識涵蓋哪些主題（不用完美，先有再調）
2. 每個主題建一個資料夾（「分支」）
3. 寫一份 `index.md`，記錄每個分支的定義和邊界

### 分支設計原則

一個好的分支需要滿足：
- **獨立性**——放不進任何現有分支
- **規模**——預期能收納 5 個以上的知識點
- **邊界清晰**——能寫出「什麼該進、什麼不該進」的規則
- **教學獨立性**——能獨立支撐至少 30 分鐘的內容

常犯的錯：
- 分支太細（每個只有 2-3 個原子）→ 用 tag 關聯就好，不用開分支
- 分支太粗（一個分支塞了 50+ 原子，內容橫跨三個不同主題）→ 拆分
- 分支邊界模糊（超過 50% 的原子同時被標記了另一個分支的 tag）→ 合併或重新定義邊界

### AI 的角色

讓 AI 看你的素材清單，建議分支結構。但最終判斷是你的——AI 不知道你的教學定位和差異化在哪。

## Phase 2：段落分類

**目標**：掃描原始素材，標記每一段「值得萃取」或「跳過」。

這一步是整個流程品質的守門員。不做分類就直接萃取，會產出大量低品質原子，事後清理的成本比事前過濾高很多。實戰數據：社群回覆的過濾率高達 87%——絕大多數是噪音（表情、打招呼、「推」「+1」）。先分類，只萃取值得的段落。

### 分類標準

逐段判斷，標記為以下三類之一：

- **萃取**——包含可獨立成立的知識點、觀點、經驗、方法。即使脫離原文脈絡，這段話本身就有教學或參考價值
- **跳過**——純社交互動、重複他人觀點、沒有資訊量的附和、行動項（「明天要做 X」）、純情緒表達
- **待定**——有潛在價值但不確定，或需要搭配上下文才能理解。待定的段落在第一輪跳過，全部處理完後回頭統一決定

### 不同素材的分類傾向

| 素材類型 | 預期萃取率 | 分類重點 |
|---------|-----------|---------|
| 社群貼文 | 70-90% | 已經過作者自我篩選，大部分值得萃取 |
| 社群回覆 | 10-15% | 絕大多數是噪音，只挑有深度解釋的 |
| 逐字稿 | 40-60% | 跳過寒暄、重複、離題段落 |
| 文章/論文 | 80-95% | 幾乎都值得，主要判斷是否跟知識庫主題相關 |
| 筆記/備忘 | 20-40% | 大量行動項和碎片想法不是知識 |
| 對話紀錄 | 15-30% | 只萃取結論和決策，跳過討論過程 |

## Phase 3：萃取成原子

**目標**：把分類為「萃取」的段落，轉成標準格式的知識原子。

### 什麼是知識原子

一個 `.md` 檔案，包含 YAML frontmatter（metadata）和一個核心論點的萃取內容。學術界稱這個過程為 Atomic Fact Decomposition——把複合資訊拆解為最小的、可獨立驗證的知識單元。

**原子是 immutable 的。** 建立後不修改。如果知識演進了（觀點改變、技術更新），建新原子取代舊的，舊原子移到 `archived/`。然後重新編譯受影響的 Wiki 頁面。這對應 Karpathy 的 `raw/` 不可修改原則——原子是我們的 raw，Wiki 隨時可以從原子重新編譯。

```yaml
---
id: branch/descriptive-slug
type: explanation | opinion | tutorial | myth-busting | case-study | comparison
depth: beginner | intermediate | advanced
source_type: post | transcript | article | note | screenshot | audio
source_ids: []
reuse_score: high | medium | low
tags: []
created: YYYY-MM-DD
---
```

### 萃取原則

1. **萃取，不是搬運**——存的是提煉後的知識，不是原文複製貼上
2. **一個原子一個核心論點**——一篇文章有三個獨立觀點，就拆成三個原子
3. **保留作者的語氣和立場**——個人知識庫的差異化來自觀點，不要寫成中性百科
4. **標記來源**——每個原子追溯到原始素材，未來更新時知道去哪找
5. **一個原子只歸一個分支**——用 tags 標記關聯主題

### 怎麼判斷「萃取」vs「搬運」

以下是同一段原文的差與好的處理方式：

**原文**（社群貼文）：
> 昨天花了三小時 debug 一個 agent，最後發現問題是 system prompt 太長，模型根本沒在看後半段。後來我把 prompt 拆成三層——角色定義、任務規則、當下 context——立刻就正常了。真的不要把所有東西塞在一個 prompt 裡面。

**差的原子（搬運）**：

```markdown
# debug agent 的經驗
昨天花了三小時 debug 一個 agent，最後發現問題是 system prompt 太長，
模型根本沒在看後半段。後來我把 prompt 拆成三層——角色定義、任務規則、
當下 context——立刻就正常了。真的不要把所有東西塞在一個 prompt 裡面。
```

→ 這就是把原文換了個標題貼上去，沒有任何提煉。

**好的原子（萃取）**：

```markdown
# System Prompt 分層架構

長 system prompt 會導致模型忽略後半段指令。解法是把 prompt 拆成三層結構：

1. **角色定義**——你是誰、核心職責
2. **任務規則**——操作約束、品質標準
3. **當下 context**——這次任務的具體資訊

分層的好處不只是避免遺漏，更讓每一層可以獨立更新。角色定義幾乎不變，
任務規則偶爾調整，context 每次都換——混在一起的話，改任何一層都要重新
檢查整個 prompt。
```

→ 提煉了核心知識點（分層架構），補充了原文沒明說但隱含的洞察（獨立更新的好處），用結構化的方式呈現。

### 什麼時候一段拆成多個原子

**判斷標準**：如果一段文字裡有兩個觀點，拿掉其中一個，另一個仍然完整成立——就該拆。

**拆的範例**：

原文：「Vibe Coding 不是不寫 code，是把精力從語法轉到架構。而且我發現用 AI 寫 code 之後，code review 變得比以前更重要，因為你要審的不是你自己寫的邏輯。」

→ 拆成兩個原子：
1. **Vibe Coding 的本質**——精力從語法轉到架構（type: explanation）
2. **AI 時代 Code Review 的轉變**——審的是 AI 的邏輯不是自己的（type: opinion）

**不拆的範例**：

原文：「Harness 的三層架構是 Persona、Rules、Context。Persona 定義角色，Rules 定義約束，Context 提供當下資訊。」

→ 不拆。三層是同一個概念的展開，拆開來每一層都不完整。

### 什麼不該變成原子

以下內容在分類階段標記為「跳過」，但如果漏進來了，萃取時也要擋掉：

- **純行動項**：「明天要改 landing page」→ 這是 todo，不是知識
- **無觀點的轉述**：「OpenAI 發佈了 GPT-5」→ 新聞，除非附帶作者的分析
- **過度依賴時效的內容**：「目前 Claude 3.5 Sonnet 最強」→ 三個月後就過時，除非在討論評估方法論
- **純情緒**：「今天超累但很有成就感」→ 日記，不是知識

### 品質判斷

`reuse_score` 怎麼標：
- **high**：獨立拿出來就能放進講座/文章，有明確觀點加支撐
- **medium**：需要搭配其他原子才能成為完整內容，但觀點有價值
- **low**：資訊性內容，可能只在特定場景用到

### 批次處理的順序建議

如果你有多種素材，建議的處理順序：

1. **公開發表過的內容先**——已經過自我篩選，品質基線高
2. **深度素材次之**——補充公開內容沒講到的技術細節
3. **私人筆記最後**——需要最多判斷，且容易產出低品質原子

先有骨架（公開內容），再補肌肉（深度素材），最後加細節（筆記）。沒有骨架就處理筆記，你會淹沒在細節裡。

### 首批校準

第一批素材（約 10-20 段）處理完後，必須人工審核。看兩件事：
1. 分類有沒有漏掉值得萃取的段落
2. 萃取的品質是不是你要的——觀點有沒有被稀釋、結構化程度夠不夠

校準完成後，後續批次就能穩定產出。跳過校準直接大量處理 = 賭運氣。

## Phase 4：品質整理

**目標**：清理萃取階段不可避免的問題。

### 檢查清單

- [ ] **去重**：同一個觀點從不同素材被萃取了多次 → 保留最完整版本，歸檔其他
- [ ] **分類校準**：原子放錯分支 → 移到正確分支，更新 frontmatter
- [ ] **膨脹分支處理**：單一分支超過 30 個原子且可自然分群 → 拆分
- [ ] **depth/reuse_score 校準**：批次萃取時 AI 可能標記不一致 → 統一標準
- [ ] **缺口分析**：每個分支的子主題分佈、深度分佈是否合理

### 缺口分析的三層方法

1. **主題完整度**——每個分支涵蓋了哪些子主題，明顯缺什麼
2. **深度梯度**——beginner/intermediate/advanced 的分佈是否合理
3. **用途對照**——你的教學產品/內容計劃需要的知識，分支是否覆蓋

缺口不代表一定要補。有些缺口是因為你在那個主題確實沒什麼好說的——那就是真實的知識邊界，不需要硬補。

### AI 的角色

缺口分析特別適合 AI 做——它擅長統計分佈、發現模式、交叉比對。但「這個缺口重不重要」的判斷是你的。

## Phase 5：外部校驗 + 統整

**目標**：確認事實準確性，為每個分支產出可讀的摘要。

### 外部校驗

用 WebSearch 或其他工具驗證：
- 技術聲明是否正確/過時
- 數據是否需要更新
- 有沒有重要的反面觀點你漏掉了

**原則**：外部校驗是佐證和補充，不是取代你的第一手觀點。別人怎麼看是參考，你怎麼看才是知識庫的核心價值。

### 分支摘要

每個分支產出一份 SUMMARY.md：
- **核心論述線**——這個分支在講什麼故事
- **原子清單**——按子主題和深度排列
- **教學路徑建議**——哪些原子適合什麼場景
- **已知缺口**——還需要補什麼

這一步需要有綜合能力的 AI 模型。便宜的模型可以做萃取和格式化，但「從 60 個原子中提煉出一條教學敘事線」需要更強的理解力。

## Phase 6：Wiki 編譯

**目標**：把零散的原子編譯成可閱讀的 Wiki 頁面。

這一步是整個流程的高光。原子是零件，Wiki 頁面是成品。

### 編譯邏輯

不是把每個原子變成一頁。是按「一個讀者想理解的主題」把相關原子組合成一篇連貫的文章。

舉例：你有 5 個原子分別講「Harness 的定義」「為什麼需要 Harness」「AI 協作的三個挑戰」「Harness vs Agent」「Harness 發佈公告」——這 5 個合成一頁「什麼是 Harness Engineering」。

### 格式約定（自動化前置條件）

以下約定讓 Lint 腳本和 index 自動生成能運作：

1. **檔名規則**——`{branch}-{topic-slug}.md`，全小寫，用連字號分隔。例：`harness-engineering-what-is-harness.md`、`context-engineering-three-layer-prompt.md`
2. **`[[wiki link]]` = 檔名去掉 `.md`**——`[[harness-engineering-what-is-harness]]` 對應 `wiki/harness-engineering-what-is-harness.md`。腳本靠這個規則檢查幽靈連結和孤立頁面
3. **每頁第一行必須是 `# 標題`**——index.md 自動生成會抓這行作為頁面標題
4. **`[[link]]` 可出現在內文任何位置**——不限於「延伸閱讀」區塊。首次提到相關概念時加連結，之後同頁再提到不重複加
5. **時效性標記用統一格式**——版本號寫 `v3.5`、日期寫 `2025-04`、避免用「目前」「最新」等模糊時效詞，改用具體日期（「截至 2025-04」）。Lint 腳本用 regex 掃這些格式檢查過時聲明

### 編譯原則

1. **按主題群組，不按原子一對一**——一頁 Wiki 通常編譯 3-8 個原子
2. **保留作者的語氣**——Wiki 不是百科全書，是有觀點的知識整理
3. **加交叉引用**——用 `[[wiki link]]` 連結相關頁面，讓知識形成網路
4. **標記來源**——頁尾列出編譯自哪些原子，方便追溯
5. **控制長度**——單頁建議 1500-2500 字。太長就拆

### Wiki 頁面結構

```markdown
# 頁面標題

開頭一段破題（為什麼這個主題重要、常見誤解是什麼）

## 小節一
[整合多個原子的內容，用教學口吻寫成連貫段落]

## 小節二
[繼續展開，自然過渡]

---

**延伸閱讀**
- [[相關頁面一]] -- 一行描述
- [[相關頁面二]] -- 一行描述

---
*編譯自 N 個知識原子：atom-a、atom-b、atom-c*
```

### 全局索引

建一份 `wiki/index.md`，列出所有 Wiki 頁面，按分類組織，每頁一行摘要。這是 LLM 查知識的入口——它不需要讀所有頁面，只要看 index 就知道去哪找。

### 變更日誌

`wiki/log.md`，append-only。每次 Ingest 新素材或更新頁面，記一筆。

### AI 的角色

Wiki 編譯是 AI 最能發揮的地方。它擅長的就是「把 5 篇零散內容合成一篇有結構的文章」。但要給它：
- 明確的語氣指示（是教學風格還是技術文件風格）
- 範例頁面（讓它看一頁你滿意的 Wiki 長什麼樣）
- 分支的 SUMMARY.md（讓它理解整體敘事線）

## 持續維護：三個核心操作

Wiki 不是建完就結束。Karpathy 定義了三個持續操作：

### Ingest（有新素材時）

1. 新素材放入 `raw/`（或你的素材來源目錄）
2. AI 讀取新素材，萃取成原子
3. 更新相關 Wiki 頁面（或建新頁面）
4. 更新 `index.md` 和 `log.md`

### Query（查知識時）

1. 查 `index.md` 定位相關頁面
2. 讀取頁面，綜合回答
3. 如果回答產生了新的有價值綜合，寫回 `wiki/syntheses/`

### Lint（定期檢查）

Lint 是讓 LLM 審計整個 Wiki 的健康度。頻率另外排程決定。

#### 執行方式

1. **讀取範圍**：LLM 讀 `index.md` + 所有 wiki 頁面（不需要讀原子，Lint 檢查的是 wiki 層的品質）
2. **逐項檢查**：
   - **矛盾偵測**——A 頁說「X 是最佳做法」，B 頁說「X 已過時」→ 標記衝突，列出兩頁路徑和矛盾段落
   - **孤立頁面**——沒有任何其他頁面的 `[[wiki link]]` 指向它 → 要嘛補連結，要嘛合併進其他頁面
   - **幽靈連結**——`[[wiki link]]` 指向不存在的頁面 → 要嘛建頁面，要嘛移除連結
   - **概念缺口**——多個頁面反覆提到某個概念但沒有獨立頁面 → 標記為候選新頁面
   - **過時聲明**——包含版本號、日期、「目前」等時效性用語的段落 → 標記需要驗證
3. **產出格式**：一份 `wiki/lint-report.md`，按嚴重度排序（矛盾 > 幽靈連結 > 孤立頁面 > 概念缺口 > 過時聲明），每項列出具體頁面和段落位置
4. **處理**：根據報告決定要修哪些 wiki 頁面，修完後更新 `log.md`

## 進階：graphify 與圖譜化知識

如果你的需求超越了 Wiki 的扁平結構，[graphify](https://github.com/safishamsi/graphify)（25k+ stars）提供了圖譜化的進路。

### graphify 做了什麼

跟 Karpathy Wiki 解決同一個問題——「有一堆原始素材，怎麼變成結構化知識」——但用知識圖譜而不是 Wiki 頁面。

技術亮點：
- **三階段管線**：AST 解析 → 轉錄提取 → LLM 語意分析。比直接丟給 LLM 讀檔案省 71.5 倍 token
- **多模態支援**：程式碼、圖片、影片、音訊都能處理
- **拓撲聚類**：用 Leiden 演算法做社區偵測，不靠 embedding 相似度——抓到的是結構關係，不是表面語意相似
- **三層信心模型**：EXTRACTED（直接從原文抽取）、INFERRED（推論關係）、AMBIGUOUS（不確定）

### 什麼時候用 graphify 而不是 Wiki

| 場景 | Wiki | graphify |
|------|------|----------|
| 個人知識整理（< 200 頁） | 最適合 | 殺雞用牛刀 |
| 大型程式碼庫理解 | 不適合 | 很適合 |
| 多模態素材（程式碼 + 文件 + 影片） | 需要先統一格式 | 原生支援 |
| 想要互動式視覺化 | 不提供 | 內建 |
| 想要人類可閱讀的知識頁面 | 核心價值 | 不是重點 |
| 跨領域知識連結探索 | 靠人工建 `[[link]]` | 自動發現 |

### 混合方案

兩者不互斥。一個可行的組合：
1. 用本文的 Phase 1-5 建立原子和分支結構
2. 用 Phase 6 編譯可讀的 Wiki 頁面（給人看）
3. 把同一批原子餵給 graphify 產出知識圖譜（給探索用）

Wiki 回答「X 是什麼」，圖譜回答「X 跟什麼有關、可能跟什麼有關」。

## 常見問題

### 「我的素材量不大，值得這樣做嗎？」

看你怎麼定義「值得」。如果你只有 10 篇筆記，直接放一個資料夾就好。但如果你有 50+ 篇素材，而且未來會持續產出，花一個下午建結構會在之後每次使用時省下翻找的時間。知識庫是複利的——越早建，之後用的每一次都是回報。

### 「一定要用 YAML frontmatter 嗎？」

不一定。YAML frontmatter 是為了讓 AI 能批次處理和統計分析。如果你的知識庫小到不需要批次操作，純文字就夠了。但建議至少標記 `type` 和 `tags`——之後要找東西的時候會感謝自己。

### 「AI 萃取的品質夠嗎？」

取決於你給的規則和範例。沒有規則的 AI 萃取 = 隨機品質。有明確規則 + 2-3 個範例 + 人工審核前幾批 = 品質穩定可接受。關鍵不是 AI 夠不夠聰明，是你有沒有把標準講清楚。

### 「Compile 和 RAG 只能選一個嗎？」

不是。小規模用 Compile（Wiki），大規模用混合方案。Compile 處理你的核心知識（高品質、穩定、值得投入整理的），RAG 處理邊緣查詢（一次性、大量、不值得人工整理的）。

### 「知識過時了怎麼辦？」

定期 Lint。技術知識建議每季度跑一次 Lint，檢查：
- 工具版本號是否還正確
- 技術趨勢有沒有變化
- 你自己的觀點有沒有演進

觀點演進不是壞事。把舊觀點標記為「已演進」，保留脈絡，不要刪除。知道自己怎麼變的也是一種知識。

## 實戰數據

以下數據來自一個真實的個人知識庫建構案例（11 個分支、625 個原子、Phase 1-F 完整走過一遍）：

| 指標 | 數值 |
|------|------|
| 原始素材 | 584 篇社群貼文 + 8,668 則回覆 + 講座教材 + 課程素材 |
| 有效萃取 | 625 個知識原子（回覆過濾率 87%——大部分回覆是噪音） |
| 分支數 | 11（從 10 開始，Phase A 拆出 1 個新分支） |
| 品質整理調整 | 16 個重分類 + 3 組去重 + 1 次分支拆分 |
| 外部校驗 | 修正 2 個過時聲明 + 補充 4 個新原子 |
| Wiki 編譯 | 每個分支的 SUMMARY.md + Wiki 頁面（3-8 個原子合成一頁） |
| 最大分支 | 101 個原子（developer-workflow） |
| 最小分支 | 23 個原子（harness-engineering）—— 晚期才系統化的概念，正常 |

### 教訓

- **回覆的價值被低估**。社群回覆常包含比貼文更深的技術細節，因為是在具體問題脈絡下的解釋。但過濾成本高（87% 是噪音）
- **批次處理的模型選擇有差異**。萃取（Phase 2-3）用中等模型就夠；統整（Phase E）和編譯（Phase 6）需要強模型，因為要從幾十個原子中提煉敘事線
- **分支設計會演化**。不要期望第一版就完美。Phase A 發現一個分支需要拆分，完全正常
- **缺口不一定要補**。缺口分析會列出「理論上應該有但沒有」的知識點。有些是真的缺，有些只是你在那個主題沒什麼好說的——後者不需要硬補
